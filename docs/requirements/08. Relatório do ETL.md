# Relatório de ETL

## Visão Geral da Arquitetura de Dados

O processo de Engenharia de Dados do SUSi foi estruturado seguindo o modelo ETL (Extract, Transform, Load) para consolidar informações fragmentadas da regulação pública de saúde em uma base única, limpa e analítica.

## Base Processada

A base final (`base.csv`) contém as seguintes variáveis estruturadas:

| Variável | Tipo de Dado | Descrição |
|----------|--------------|-----------|
| `appointment_id` | String (UUID) | Identificador único do agendamento no sistema de regulação |
| `patient_id` | String (UUID) | Identificador único anonimizado do paciente |
| `requesting_cne` | String | Código CNES da unidade de atenção primária que solicitou a vaga |
| `executing_cnes` | String | Código CNES da unidade especializada onde o procedimento ocorrerá |
| `request_date` | Datetime | Data em que o pedido foi inserido no sistema de regulação |
| `appointment_date` | Datetime | Data marcada para a realização da consulta/exame |
| `procedure` | String | Descrição padronizada do procedimento (ex: CONSULTA CARDIOLOGIA) |
| `status` | String | Situação final do agendamento (ATENDIDO, FALTA, CANCELADO, AGUARDANDO) |
| `distance_km` | Float | Distância calculada em quilômetros entre o CEP do paciente e a unidade executante |
| `wait_time_days` | Integer | Diferença em dias entre `request_date` e `appointment_date` |
| `negative_outcome` | Integer (0/1) | Variável binária (Target) onde 1 indica ineficiência (FALTA ou CANCELADO) |
| `municipality` | String | Município onde a unidade executante está localizada |
| `facility_type` | String | Classificação do estabelecimento (ex: HOSPITAL GERAL, POSTO DE SAUDE) |

## Justificativa das Decisões de Limpeza

Durante a fase de Transformação, a base bruta apresentava diversas anomalias características de dados reais do SUS. As seguintes decisões programáticas foram aplicadas utilizando a biblioteca `pandas`:

### Tratamento de Valores Ausentes (NaNs)
- Problema: A coluna `status` possuía cerca de 5% de valores nulos (falhas de preenchimento na ponta). A coluna `distance_km` possuía 2% de falhas de geolocalização.
- Risco Analítico: Modelos de Machine Learning (como Random Forest) falham ao processar matrizes com valores nulos.
- Solução Programática: 
  - Para `status`, utilizamos `.fillna('UNKNOWN')` para preservar o registro sem assumir um desfecho.
  - Para `distancia_km`, aplicamos imputação estatística pela mediana `.fillna(median_dist)`, evitando que outliers distorcessem a distribuição espacial.

### Correção de Inconsistências e Erros de Digitação
- Problema: O status de absenteísmo apresentava erros de digitação sistêmicos (ex: 'NO_SHW' em vez de 'NO_SHOW').
- Risco Analítico: A ineficiência seria subnotificada, pois o algoritmo interpretaria 'NO_SHW' como uma categoria distinta.
- Solução Programática: Utilização do método `.replace({'NO_SHW': 'NO_SHOW'})` para padronização categórica.

### Feature Engineering
- Problema: A base original não possuía a métrica de "tempo de espera", apenas datas brutas, o que impede análises de correlação diretas.
- Solução Programática: Criação da coluna `wait_time_days` através de operações vetoriais de data (`dt.days`).
- Criação da Variável Alvo: Para viabilizar a modelagem preditiva futura, criamos a variável `negative_outcome`, agregando 'FALTA' e 'CANCELADO' como indicadores de falha do sistema.

### Merge
- Problema: A base do SISREG (agendamentos) possuía apenas o código CNES, sem dados descritivos da unidade, dificultando análises geográficas.
- Solução Programática: Execução de um `pd.merge()` (Left Join) entre a base de agendamentos e o cadastro do CNES, enriquecendo os registros com o `municipality` e o `facility_type`.

## Conclusão

O script de ETL (`ETL.py`) demonstrou que é possível automatizar a transformação de dados caóticos em um artefato analítico estruturado. A base resultante (`base.csv`) está íntegra, livre de valores nulos críticos e pronta para análise exploratória e modelagem estatística.
