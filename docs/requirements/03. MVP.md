# Escopo e MVP

## Visão do Produto (MVP Analítico)

O Produto Mínimo Viável (MVP) do SUSi é um Pipeline de Engenharia de Dados e Análise Preditiva. 

O objetivo é provar a viabilidade técnica de extrair, transformar e analisar dados públicos fragmentados do SUS para evidenciar as falhas estruturais na jornada do paciente.

## Escopo do MVP

O MVP será composto por três camadas principais:

### Camada 1: Engenharia de Dados (ETL)
- Extração: Coleta de dados de fontes públicas (arquivos CSV do DATASUS/TabNet, dados geográficos do IBGE e cadastro de unidades do CNES).
- Transformação: Limpeza de dados sujos (tratamento de valores nulos, padronização de formatos de data, remoção de duplicatas e junção de bases heterogêneas) utilizando a biblioteca `pandas` em Python.
- Carga: Estruturação dos dados limpos em um banco de dados relacional local (PostgreSQL) para consultas otimizadas.

### Camada 2: Análise e Modelagem Preditiva
- Análise Exploratória (EDA): Scripts em Python para calcular estatísticas descritivas e identificar correlações (ex: distância vs. taxa de não comparecimento).
- Modelo de Machine Learning: Treinamento de um algoritmo de classificação (Random Forest) utilizando a biblioteca `scikit-learn` para prever o risco de desfechos negativos na jornada do paciente com base em variáveis estruturais.

### Camada 3: Visualização (Dashboard)
- Interface: Um dashboard interativo desenvolvido em `Streamlit`.
- Funcionalidades: Visualização gráfica dos gargalos regulatórios, permitindo ao usuário (simulando um gestor de saúde) filtrar dados por região e especialidade médica.

## Fora do Escopo
- Desenvolvimento de aplicativos mobile para pacientes.
- Integração em tempo real com o SISREG via API (recortes de dados estáticos para prova de conceito).
- Módulos de agendamento ou cancelamento de consultas.
- Hospedagem do banco de dados em nuvem (arquitetura local/containerizada para o MVP).

## Requisitos Técnicos

### Requisitos Funcionais (RF)
- RF01: O sistema deve ser capaz de ler arquivos CSV brutos do DATASUS e CNES.
- RF02: O script de ETL deve identificar e tratar valores nulos e inconsistentes nas bases de dados.
- RF03: O sistema deve consolidar as informações em um banco de dados relacional.
- RF04: O modelo preditivo deve classificar registros de agendamento com uma probabilidade de desfecho negativo.
- RF05: O dashboard deve apresentar gráficos de barras e mapas de calor com base nos dados processados.

### Requisitos Não Funcionais (RNF)
- RNF01: A linguagem de programação base para todo o pipeline de dados deve ser Python 3.11+.
- RNF02: O tempo de execução do script de ETL para o recorte de dados do MVP não deve exceder 5 minutos.
- RNF03: O código deve seguir as boas práticas de documentação e legibilidade (PEP 8).
- RNF04: O dashboard deve ser responsivo e acessível via navegador web padrão.
